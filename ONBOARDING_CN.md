# Cube-Studio 新人上手指南（代码库导览）

## 1. 代码库整体结构（先建立地图）

> 目标：先知道“什么目录负责什么”，再深入某个模块。

- `myapp/`：后端主应用（Flask + Flask-AppBuilder），包含模型、视图、任务、工具、模板、静态资源。核心业务代码都在这里。
- `myapp/frontend/`：主前端（React + TypeScript）。平台主 UI 在这里维护。
- `myapp/vision/` 与 `myapp/visionPlus/`：两个流程编排前端（分别偏 ML pipeline 与 ETL pipeline）。
- `job-template/`：任务模板生态（训练/处理/服务化等可插拔模板），可扩展性入口。
- `images/`：平台相关基础镜像、notebook 镜像、serving 镜像与构建脚本。
- `install/docker/`：本地开发/调试与单机部署入口（docker-compose、后端/前端镜像构建）。
- `install/kubernetes/`：集群部署相关清单与脚本（多集群、依赖组件、运维脚本）。

## 2. 运行链路（理解“系统怎么跑起来”）

### 2.1 后端启动骨架

- `myapp/__init__.py` 创建 Flask app、加载配置、初始化数据库连接、缓存、迁移、权限体系与 AppBuilder。
- 在 `MyIndexView` 中，登录后默认跳转到 `/frontend/`，说明前后端是“后端鉴权 + 前端主界面”的组合模式。
- `myapp/views/route.py` 提供基础存活探针：`/health`、`/healthcheck`、`/ping`。

### 2.2 配置中心

- `install/docker/config.py` 是核心配置样例（认证方式、语言、上传目录、CORS、日志、Web 服务端口等）。
- `myapp/config.py` 为空，表明实际部署常通过环境变量 `MYAPP_CONFIG` 指向外部配置模块。

### 2.3 异步任务与平台自动化

- `myapp/tasks/celery_app.py` 初始化统一 Celery app。
- `myapp/tasks/async_task.py` 承担平台异步动作：如 notebook/docker 镜像提交检查、推理服务滚动升级与通知。
- 这说明很多“页面上一键操作”最终会落到异步任务 + K8s API 调用。

## 3. 新人必须优先理解的 8 件事

1. **多租户与权限模型**：用户/角色/项目组是资源隔离基础。
2. **资源组织方式**：项目组、资源组、集群、命名空间之间如何映射。
3. **任务模板机制**：新算法能力通常以 `job-template` 扩展，而非改平台底层。
4. **前后端边界**：后端负责鉴权、元数据、调度与编排；前端负责交互与可视化。
5. **K8s 是真实执行面**：平台只是控制面，训练/推理实际都在集群里。
6. **异步思维**：很多动作是“提交任务 -> Celery 异步轮询/回调 -> 状态落库 -> 前端展示”。
7. **配置驱动**：认证、跨域、日志、对象存储、域名、集群接入几乎都通过配置项控制。
8. **发布路径**：本地（docker-compose）-> 镜像构建 -> 集群部署（k8s manifests/helm-like 资源）。

## 4. 推荐学习路径（2~4 周）

### 第 1 周：跑通与认目录

- 按 `install/docker/README.md` 跑通本地 docker-compose。
- 打开平台后完成一次：登录 -> 建项目组 -> 创建 notebook -> 提交一个简单任务模板。
- 目标：建立“页面操作与后端对象”的映射。

### 第 2 周：后端主干

- 重点阅读：
  - `myapp/__init__.py`（应用初始化）
  - `myapp/models/`（数据模型）
  - `myapp/views/`（页面/API 行为）
  - `myapp/tasks/`（异步任务）
- 选一个功能（如 notebook 或 inference service）画出时序图：前端操作 -> 后端 view -> model -> k8s -> 回写状态。

### 第 3 周：前端与编排

- 在 `myapp/frontend/` 完成一个小改动（如新增列表字段或筛选项）。
- 阅读 `myapp/vision` / `myapp/visionPlus` 的启动方式与路由，理解 pipeline 可视化编辑器的数据结构。

### 第 4 周：扩展能力

- 从 `job-template/` 复制一个现有模板，改造为“你业务里的最小可用模板”。
- 补齐 readme、参数 schema、镜像构建脚本，完成在页面注册并运行。

## 5. 实操建议（避免新人常见坑）

- **先做“可观测”再改逻辑**：先掌握日志位置、任务状态字段、Pod 状态查看命令。
- **先改模板后改平台**：80% 的业务需求可用模板实现，成本更低。
- **改一处，走一条完整链路**：不要只看页面成功，要确认 DB 状态和 K8s 资源一致。
- **给每个改动配回滚点**：镜像 tag、配置项、模板版本都要可回退。
- **优先小步提交**：一个 PR 只解决一个问题，便于评审和定位风险。

## 6. 给导师/带教的建议（如何带新人更高效）

- 第一次讲解建议按“业务场景”讲，不要按目录逐个文件讲。
- 让新人在第一周内完成 1 个真实任务（哪怕很小），比看文档更有效。
- 让新人维护 1 个模板 + 1 个页面字段，会快速打通全栈理解。
